# Observability

This document details logging patterns, metrics, and alerting rules for monitoring real-time chat operations.

---

## 1. Logging Strategy

Logging uses **SLF4J** and **Logback**.

### Rules
1.  **Mask Sensitive Data**: Passwords and cleartext tokens must be masked.
2.  **Trace Correlation**: In WebSocket configurations, map the user session ID or trace ID in the STOMP message headers to MDC (Mapped Diagnostic Context) so that all logs generated by a message broadcast share the same context trace ID.

---

## 2. Core Real-Time Metrics

To monitor real-time message operations, Prometheus tracks the following metrics via Micrometer:

| Metric Name | Type | Description | Alert Condition |
| :--- | :--- | :--- | :--- |
| `websocket.sessions.active` | Gauge | Number of active connected WebSocket channels. | Sudden drop of >50% (signals network outage). |
| `message.send.rate` | Counter | Number of messages published to `/app/chat.sendMessage`. | Spike > 10,000/sec (potential spam/bot attack). |
| `message.processing.time` | Timer | Latency to save to database and broadcast to broker. | `p99 latency > 200ms` |
| `websocket.errors` | Counter | Socket connection errors, closed states. | Rate > 5% of total sessions. |

---

## 3. Alerts & Dashboard Panels

Set up dashboards showing:
1.  **Active Sessions vs. Network I/O**: High session counts with low I/O signals stale connections, indicating a need to optimize heartbeat rates.
2.  **Database Connection Wait Times**: Monitor pool starvation during high message volume spikes.
3.  **STOMP Frame Errors**: Track exceptions thrown in `ChannelInterceptor` (e.g. `AccessDeniedException` count), indicating potential malicious subscription attempts.
